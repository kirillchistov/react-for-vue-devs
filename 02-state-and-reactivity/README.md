# Модуль 02. Состояние: явные обновления вместо автослежения

## Смена со старшим инженером Хуксом

«Витрину вы построили, теперь нужно, чтобы с неё что-то можно было забрать. Добавляем кнопку «Добавить» на карточку и панель конструктора сбоку — кладёте туда ингредиенты, видите сумму, убираете, что передумали.

На Proxy Drive вы бы завели `reactive([])` и просто пушили в него — движок сам бы заметил. Здесь так не будет. Я слежу за состоянием станции лично, и любое изменение проходит только через официальный канал — `useState`. Если вы измените что-то в обход него, для меня как будто ничего не произошло — экран не обновится, а ошибки не будет. Это самый частый способ, которым инженеры с вашим прошлым простреливают себе ногу в первую неделю».

## Идея модуля: React не отслеживает мутации

В Vue `reactive()` оборачивает объект в Proxy: любое присваивание или мутация (`.push`, `obj.field = x`) автоматически перехватывается и вызывает перерисовку зависимых частей шаблона. Это и есть автослежение — вам не нужно ничего явно «сообщать».

В React ничего не оборачивается в Proxy. `useState` даёт вам пару: текущее значение и функцию-сеттер. React решает, перерисовывать компонент или нет, по одному критерию — **изменилась ли ссылка**, которую вы передали в сеттер, по сравнению с предыдущим рендером. Мутация объекта или массива на месте ссылку не меняет — и, значит, для React ничего не произошло, даже если по факту данные внутри уже другие.

```js
// Vue: mutation — это и есть API
const cartItems = reactive([]);

function addItem(ingredient) {
  cartItems.push(ingredient); // Proxy сам заметил, шаблон сам перерисовался
}
```

```jsx
// React: мутация ничего не даёт
const [cartItems, setCartItems] = useState([]);

function addItem(ingredient) {
  cartItems.push(ingredient); // массив изменился, но ССЫЛКА на него — нет
  setCartItems(cartItems);     // та же ссылка → React не видит повода перерисовать
}

// Правильно — новая ссылка на каждое обновление:
function addItem(ingredient) {
  setCartItems([...cartItems, ingredient]);
}
```

Тот же принцип для объектов: `setUser({ ...user, name: 'Новое имя' })`, а не `user.name = 'Новое имя'; setUser(user)`. Это не стилистическое предпочтение — это единственный способ, которым React вообще узнаёт, что что-то изменилось.

## Инспектор Тай уже здесь

Новый тип в этом модуле — `BurgerItem`:

```ts
export interface BurgerItem {
  uid: string;
  ingredient: Ingredient;
}
```

Казалось бы, зачем не хранить в конструкторе просто `Ingredient[]`? Затем, что один и тот же ингредиент можно добавить дважды (два одинаковых соуса — обычное дело), а `ingredient.id` у обоих будет одинаковым. Если использовать `ingredient.id` как React `key` в списке конструктора — вы получите ровно тот баг, который разбирали в шпаргалке модуля 00 и в вопросе 9 входной диагностики: React перепутает состояние строк местами. `uid` — это идентификатор конкретного *слота* в конструкторе, а не ингредиента, и генерируется он в момент добавления (`crypto.randomUUID()`), а не берётся из каталога.

## Проверь себя

<details>
<summary>1. Почему <code>state.push(x); setState(state)</code> не работает, а <code>setState([...state, x])</code> — работает?</summary>

React решает, перерисовывать компонент или нет, сравнивая ссылку на state между рендерами, а не заглядывая внутрь массива. `push` мутирует массив на месте — ссылка остаётся той же, и `setState(state)` передаёт ту же самую ссылку, которая уже была. Spread (`[...state, x]`) создаёт новый массив с новой ссылкой — именно это React и умеет заметить.
</details>

<details>
<summary>2. В чём разница между <code>ingredient.id</code> и <code>item.uid</code> в этом модуле?</summary>

`ingredient.id` идентифицирует позицию в каталоге — он одинаковый для всех экземпляров одного и того же ингредиента. `uid` идентифицирует конкретный слот в конструкторе — он создаётся заново при каждом добавлении, поэтому даже два одинаковых ингредиента в бургере получат разные `uid`. Без этого разделения нельзя было бы корректно удалить «второй такой же соус», не задев первый.
</details>

<details>
<summary>3. Почему для удаления элемента по <code>uid</code> используют <code>.filter()</code>, а не <code>.splice()</code>?</summary>

`.splice()` мутирует массив на месте (и ещё возвращает не то, что вы ожидаете — удалённые элементы, а не новый массив). `.filter()` — чистая функция: она возвращает совершенно новый массив, ничего не трогая в исходном. Для React это единственный вариант, который реально приводит к перерисовке.
</details>

<details>
<summary>4. Что бы изменилось, если бы конструктор хранил не <code>BurgerItem[]</code>, а просто <code>Ingredient[]</code>?</summary>

Стало бы невозможно надёжно удалить «именно этот» экземпляр повторяющегося ингредиента — `.filter((i) => i.id !== targetId)` удалил бы сразу все совпадения по `id`, а не один конкретный слот. Пришлось бы городить удаление по индексу, а это прямая дорога к багам с `key={index}` из модуля 01.
</details>

<details>
<summary>5. Компонент вызывает <code>setState</code> с точно таким же значением, какое уже было. Перерисуется ли компонент?</summary>

Если это тот же самый примитив (число, строка, `true`/`false`) — нет, React сравнивает примитивы по значению и пропускает лишний рендер. А вот если это новый объект или массив с тем же содержимым (например, вы каждый раз строите `{ ...user }` без реальных изменений) — да, перерисуется: React сравнивает объекты по ссылке, а не по глубокому содержимому, и новый объект — это всегда «другая» ссылка, даже если поля внутри идентичны.
</details>
