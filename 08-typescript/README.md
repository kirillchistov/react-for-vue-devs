# Модуль 08. Продвинутый TypeScript

## Смена с инспектором Таем

«Обычно я захожу в конце смены и придираюсь к типам, которые вы уже написали. Сегодня — наоборот, смена целиком моя. Типизация с первого модуля прикрыла базовые случаи, но три инструмента мы ещё не трогали: типы, которые подстраиваются под то, что в них передали (дженерики); типы, которые описывают ровно те состояния, что реально бывают, и ни одним больше (дискриминированные объединения); и типы, собранные из уже существующих, а не написанные с нуля (утилитарные типы).

Ничего из этого не про синтаксис ради синтаксиса. У `useIngredients` до сегодняшнего дня было два независимых поля — `ingredients` и `status` — и ничто не запрещало оказаться в состоянии `status: 'error'` с полным списком ингредиентов внутри, оставшимся от предыдущей удачной загрузки. Технически возможно, семантически — ерунда. Почините это раз — и такой баг станет невозможен по конструкции, а не по соглашению между разработчиками».

## Идея модуля

### Дженерики: тип, который подстраивается под данные

Хук `useLocalStorage` из модуля 04 (в дополнительном примере) работал только со строками. Реальная `localStorage` не различает типы — она хранит строки, и сериализация/десериализация нужна для чего угодно: чисел, объектов, массивов. Дженерик-параметр `<TValue>` описывает это один раз:

```ts
function useLocalStorage<TValue>(key: string, initialValue: TValue): [TValue, (value: TValue) => void] {
  // ...
}

const [count, setCount] = useLocalStorage<number>('count', 0);
const [order, setOrder] = useLocalStorage<{ orderNumber: number } | null>('last-order', null);
```

Один и тот же хук, а `count` типизирован как `number`, `order` — как объект с `orderNumber`. TypeScript выводит это из того, что вы передали, а не угадывает и не требует `any`.

### Дискриминированные объединения: невозможные состояния — невозможны

```ts
// Так было: два поля независимо друг от друга
interface Old {
  ingredients: TIngredient[];
  status: 'loading' | 'ready' | 'error';
}

// Так стало: одно поле решает, какие ещё данные вообще существуют
type TAsyncState<TData, TError = string> =
  | { status: 'loading' }
  | { status: 'ready'; data: TData }
  | { status: 'error'; error: TError };
```

Ключевое отличие: чтобы прочитать `data`, TypeScript заставит вас сначала проверить `state.status === 'ready'` — без этой проверки `state.data` попросту не существует с точки зрения типов, обращение к нему — ошибка компиляции, а не баг, который вылезет в проде. Поле `status` — «дискриминант»: по его значению TypeScript сам понимает, какой именно вариант объединения перед вами, и сужает тип автоматически, без `as`.

### Утилитарные типы: собрать тип из уже существующего

`Record<TIngredientType, string>` вы использовали ещё в модуле 01, даже не называя это утилитарным типом. В этом модуле — `Pick`:

```ts
type TLastOrder = Pick<TOrderResponse, 'orderNumber'>;
// эквивалент { orderNumber: number }, но выведен из TOrderResponse,
// а не продублирован вручную — если в TOrderResponse изменится тип
// orderNumber, TLastOrder обновится сам, без правок в двух местах.
```

## Инспектор Тай уже здесь

(в этот раз — буквально с самого начала). Взгляните на сигнатуру:

```ts
export function useLocalStorage<TValue>(
  key: string,
  initialValue: TValue,
): [TValue, (value: TValue) => void]
```

Возвращаемый тип — кортеж (`[TValue, ...]`), а не объект `{ value, setValue }`. Это осознанное решение: `useState` возвращает кортеж именно затем, чтобы вызывающий код мог назвать элементы как угодно (`const [count, setCount] = ...`) — с объектом пришлось бы либо мириться с фиксированными именами полей, либо переименовывать через деструктуризацию (`{ value: count, setValue: setCount }`). Раз вы копируете паттерн `useState`, стоит скопировать и эту деталь.

## Проверь себя

<details>
<summary>1. Компонент получил <code>state: TAsyncState&lt;TIngredient[]&gt;</code> и написал <code>state.data.length</code> без предварительной проверки <code>state.status</code>. Что произойдёт?</summary>

Ошибка компиляции, а не рантайм-баг. TypeScript не может гарантировать, что `state.data` существует, пока не сужен тип через проверку дискриминанта (`if (state.status === 'ready')`). Без такой проверки `data` не входит в тип `state` вообще — обратиться к нему просто нельзя, каким бы ни было реальное значение в рантайме.
</details>

<details>
<summary>2. Зачем нужен параметр <code>TError = string</code> в <code>TAsyncState&lt;TData, TError = string&gt;</code>, если во всём проекте используется только <code>string</code>?</summary>

`= string` — значение по умолчанию для параметра-типа, тот же смысл, что у значений по умолчанию для обычных параметров функции. `TAsyncState<TIngredient[]>` эквивалентно `TAsyncState<TIngredient[], string>` — вам не нужно писать второй аргумент, пока он совпадает со значением по умолчанию. Если однажды понадобится состояние с более структурированной ошибкой (например, `{ code: number; message: string }`), вы просто напишете `TAsyncState<TIngredient[], { code: number; message: string }>`, не трогая само определение типа.
</details>

<details>
<summary>3. Чем <code>Pick&lt;TOrderResponse, 'orderNumber'&gt;</code> лучше, чем просто написать <code>{ orderNumber: number }</code> руками?</summary>

Результат одинаковый сегодня, но `Pick` держит связь с источником: если `TOrderResponse.orderNumber` когда-нибудь поменяет тип (например, на `string`, если заказы начнут нумероваться иначе), `TLastOrder` обновится автоматически. Ручной дубликат пришлось бы синхронизировать вручную в двух местах — а если забыть, типы разойдутся молча, без единой ошибки компиляции.
</details>

<details>
<summary>4. В <code>Select&lt;TItem&gt;</code> дженерик-параметр не указан явно ни в одном месте, где компонент используется. Как TypeScript узнаёт, что <code>TItem</code> — это <code>CrewMember</code>, а не <code>string</code>?</summary>

Из аргумента `items`, переданного в конкретном месте использования — точно так же, как обычная функция `function identity<T>(x: T): T` не требует явного `identity<number>(5)`, если `5` и так однозначно `number`. TypeScript выводит дженерик-параметры из типов переданных аргументов везде, где это возможно, и требует явного указания только в неоднозначных случаях.
</details>

<details>
<summary>5. Почему <code>useLocalStorage</code> возвращает кортеж <code>[value, setValue]</code>, а не объект <code>{ value, setValue }</code>?</summary>

Чтобы вызывающий код мог свободно переименовать элементы через деструктуризацию массива: `const [count, setCount] = useLocalStorage(...)`, `const [theme, setTheme] = useLocalStorage(...)` — оба вызова используют один и тот же хук, но называют результат так, как удобно в конкретном месте. С объектом пришлось бы либо жить с фиксированными именами полей `value`/`setValue` везде, либо каждый раз писать `{ value: count, setValue: setCount }`. Ровно поэтому так же устроен `useState`.
</details>
