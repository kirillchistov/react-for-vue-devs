# Модуль 07. Формы и работа с сервером

## Смена с энсином Формом

«Заявки на станции без меня никуда не уходят. Каждую я проверяю сам — если что-то не так, назад, на доработку, никаких исключений. Сегодня учу вас делать то же самое: форма входа не должна отправляться, пока в ней ерунда, а собранный заказ должен явно сказать, получилось его оформить или нет — не молчать неопределённое время.

На Proxy Drive у вас был `v-model` — одна директива, и поле само синхронизировано с данными в обе стороны. Здесь придётся собрать это вручную, из пары `value` + `onChange`. Непривычно первые два раза, потом руки сами вспоминают».

## Идея модуля: `v-model` — это directive-sugar, а не примитив

В Vue `v-model="email"` — синтаксический сахar, который компилятор Vue разворачивает примерно в `:value="email" @input="email = $event.target.value"`. Двусторонняя связь — часть языка шаблонов.

В React такого сахара нет вообще, потому что нет отдельного языка шаблонов, который мог бы его разворачивать. Контролируемый инпут — это ровно то, во что Vue разворачивает `v-model`, только вы пишете это явно сами:

```vue
<!-- Vue -->
<input v-model="email" />
```

```jsx
// React — то же самое, но без сахара
<input value={email} onChange={(e) => setEmail(e.target.value)} />
```

Как только это осознано, дальше — просто механика: несколько `useState`, несколько `onChange`, и функция-валидатор, которая решает, можно отправлять форму или нет.

### Асинхронная отправка — тот же паттерн, что и `useEffect` в модуле 03

Отправка заказа — это тот же `fetch`-подобный паттерн загрузки данных, только инициированный кликом, а не монтированием компонента: `pending` → `success`/`error`, с той разницей, что здесь нет `useEffect` — запрос стартует из обработчика события, а не при рендере.

```jsx
const handleSubmit = () => {
  setStatus('pending');
  submitOrder(ids)
    .then((response) => {
      setOrderNumber(response.orderNumber);
      setStatus('success');
      dispatch(clearItems()); // и очищаем то самое Redux-состояние из модуля 05
    })
    .catch(() => setStatus('error'));
};
```

## Инспектор Тай уже здесь

```ts
export interface LoginFormValues {
  email: string;
  password: string;
}

interface LoginFormProps {
  onSubmit: (values: LoginFormValues) => void;
}
```

Типизированный объект пропсов формы — тот же принцип, что типизированный результат хука в модуле 04: контракт компонента виден из одной сигнатуры, без необходимости читать реализацию. В Vue аналог — `defineProps<LoginFormValues>()`; в React это тот же самый TS-интерфейс, что вы используете для любой другой функции.

## Проверь себя

<details>
<summary>1. Почему <code>&lt;input value={email} /&gt;</code> без <code>onChange</code> «зависает» и не даёт печатать?</summary>

Как только вы передали `value`, React считает поле полностью управляемым (controlled) — то, что показывает поле, всегда равно `email` из state, а не тому, что физически напечатано. Без `onChange` state никогда не обновляется, поэтому `value` остаётся прежним, и на каждое нажатие клавиши React возвращает поле обратно к старому значению.
</details>

<details>
<summary>2. В чём разница между валидацией «по попытке отправить» (как в LoginForm) и «по потере фокуса» (как в дополнительном примере)?</summary>

В LoginForm ошибки не показываются, пока пользователь не попробовал отправить форму хотя бы раз (`touched` становится `true` только в `handleSubmit`) — это меньше раздражает при первом заполнении. Валидация по `onBlur` показывает ошибку сразу, как только пользователь покинул поле, даже если он ещё не пытался отправить форму целиком — репортит проблему раньше, но может показаться навязчивой на самом первом поле формы.
</details>

<details>
<summary>3. Зачем <code>OrderSubmit</code> дизейблит кнопку и во время <code>pending</code>, и когда <code>items.length === 0</code>?</summary>

Это два разных повода для одного и того же UI-состояния «сейчас нельзя отправить»: пустой конструктор — нечего отправлять; `pending` — запрос уже летит, повторный клик отправил бы дублирующий заказ до того, как придёт ответ на первый. Оба условия объединены в одном `disabled`, потому что пользователю не нужно знать разницу — ему достаточно того, что кнопка сейчас неактивна.
</details>

<details>
<summary>4. Почему после успешной отправки заказа вызывается <code>dispatch(clearItems())</code>, а не просто <code>setStatus('success')</code>?</summary>

`OrderSubmit` управляет только своим собственным статусом (idle/pending/success/error) — это локальное состояние компонента. Содержимое корзины (`items`) живёт в Redux (модуль 05) и видно ещё и шапке, и `BurgerConstructor`. Раз заказ подтверждён, корзину нужно очистить именно там, где она хранится по-настоящему, — через тот же `dispatch`, которым в неё добавляли ингредиенты.
</details>

<details>
<summary>5. Функция <code>isValidEmail</code> вынесена в <code>utils/validation.ts</code>, а не написана прямо внутри <code>LoginForm</code>. Почему это удобно, помимо переиспользования?</summary>

Чистую функцию без React (без хуков, без JSX) тестировать в разы проще: `isValidEmail('x')` — просто вызов и проверка результата, без `render`, без `fireEvent`, без DOM вообще. Тесты `utils/validation.test.ts` в этом модуле работают в разы быстрее и надёжнее, чем если бы та же логика была спрятана внутри компонента и проверялась только через рендер формы целиком.
</details>
