# Модуль 04. Кастомные хуки и композиция вместо миксинов

## Смена с лейтенантом Композом

«Не пишите одну и ту же процедуру дважды — вот и весь мой принцип. Загрузка ингредиентов и сборка бургера у вас уже работают, но вся логика свалена внутри `App`. Разложите её по полочкам: то, что относится к загрузке склада — в один хук, то, что относится к конструктору — в другой. `App` должен читаться как оглавление, а не как инструкция по сборке.

И заодно — окно с деталями ингредиента. На Proxy Drive вы бы прокинули именованный слот. Здесь слотов нет — есть `children`, и это ровно то же самое: компонент, которому всё равно, что именно у него внутри».

## Идея модуля: `children` вместо слотов, хуки вместо миксинов

### `children` — это просто проп

В Vue именованные и безымянные слоты — часть языка шаблонов: `<slot />`, `<slot name="header" />`. В React такого отдельного механизма нет — есть `children`, обычный проп, значение которого само является JSX.

```vue
<!-- Vue: слот -->
<Modal>
  <IngredientDetails :ingredient="ingredient" />
</Modal>
```

```jsx
// React: children — просто проп
<Modal onClose={handleClose}>
  <IngredientDetails ingredient={ingredient} />
</Modal>
```

`Modal` в этом модуле ничего не знает про ингредиенты — она умеет только показать оверлей, кнопку закрытия и то, что ей передали в `children`. Это и есть композиция: сложное поведение собирается из простых частей, а не встраивается директивами внутрь одного компонента.

### Кастомные хуки — это про переиспользование логики, а не разметки

В Vue Composition API миксины (Options API) уже были в прошлом, и composables (`useSomething()` функции) — прямой аналог кастомных хуков React. Если вы застали Vue 3 Composition API — это единственное место в курсе, где интуиция переносится почти без изменений.

Разница в деталях: composable может возвращать `reactive`-объект, за которым Vue следит через Proxy. Хук возвращает обычные значения (плюс функции-сеттеры), и всё, что их меняет, идёт через `useState`/`useReducer` внутри самого хука — никакого автослежения, те же правила, что и в компоненте.

```ts
// Хук — обычная функция, которая сама использует useState/useEffect
function useIngredients() {
  const [ingredients, setIngredients] = useState<TIngredient[]>([]);
  const [status, setStatus] = useState<TCatalogStatus>('loading');

  useEffect(() => {
    // тот же эффект, что раньше был в App
  }, []);

  return { ingredients, status };
}
```

Единственное жёсткое правило, которого нет у composables: хуки можно вызывать только на верхнем уровне компонента или другого хука — не внутри `if`, циклов или вложенных функций. React связывает состояние хука с порядком его вызова между рендерами, и нарушение порядка ломает эту связь.

## Инспектор Тай уже здесь

Тип результата хука описан явно, отдельным интерфейсом:

```ts
interface UseIngredientsResult {
  ingredients: TIngredient[];
  status: TCatalogStatus;
}

export function useIngredients(): UseIngredientsResult { ... }
```

В Vue composable обычно просто возвращает `{ ...refs объекты }`, и TypeScript выводит тип автоматически из тела функции. Для хуков это тоже работает, но явный интерфейс результата — такая же практика, как явная типизация пропсов компонента: он документирует контракт хука прямо в сигнатуре, а не заставляет читателя лезть в реализацию, чтобы понять, что хук возвращает.

## Проверь себя

<details>
<summary>1. Чем <code>children</code> в React принципиально отличается от именованного слота в Vue?</summary>

Ничем принципиальным для дефолтного слота — оба передают готовую разметку внутрь компонента-обёртки. Разница начинается с именованных слотов: у `children` нет built-in механизма «нескольких именованных мест». Это эмулируют несколькими обычными пропами, каждый из которых принимает JSX (например, `<Modal header={<X/>} footer={<Y/>}>`), а не отдельным синтаксисом.
</details>

<details>
<summary>2. Можно ли вызвать <code>useIngredients()</code> внутри <code>if (условие)</code>?</summary>

Нет. React связывает состояние каждого хука с порядковым номером его вызова относительно других хуков в этом же компоненте между рендерами. Если хук вызывается в одном рендере и не вызывается в другом, порядок сдвигается, и React либо свяжет не то состояние не с тем хуком, либо выбросит ошибку.
</details>

<details>
<summary>3. Хук возвращает <code>{ ingredients, status }</code>, но не возвращает функцию для их изменения снаружи. Компонент, который его вызвал, может напрямую поменять <code>ingredients</code>?</summary>

Нет — и это осознанное решение инкапсуляции. `setIngredients` существует только внутри хука и наружу не передаётся. Компонент может только читать результат и ждать, пока хук сам обновит его (например, после следующего запроса). Это тот же принцип, что и с пропсами: снаружи — только то, что явно предоставлено.
</details>

<details>
<summary>4. Зачем в <code>IngredientCard</code> понадобился <code>event.stopPropagation()</code> на кнопке «Добавить»?</summary>

Клик по кнопке — это одновременно и клик по родительскому элементу (событие всплывает по DOM-дереву снизу вверх). Без остановки всплытия клик по кнопке «Добавить» вызвал бы сразу оба обработчика: `onAdd` на кнопке и `onSelect` на карточке — то есть ингредиент добавился бы в конструктор, а заодно открылась бы модалка с его деталями.
</details>

<details>
<summary>5. Два разных компонента вызывают один и тот же <code>useLocalStorage('key', '')</code> с одинаковым ключом. Они будут делить одно и то же состояние?</summary>

Нет. Каждый вызов хука создаёт свой собственный, независимый `useState` внутри — компоненты не «подписываются» на общий источник просто потому, что использовали одну и ту же функцию-хук с одинаковыми аргументами. Значения синхронизируются с `localStorage`, но не друг с другом напрямую: если один компонент изменит значение, другой узнает об этом только при следующем собственном чтении (например, при перемонтировании), а не сразу и реактивно. Для «живого» общего состояния между произвольными компонентами нужен инструмент из следующего модуля — Context или Redux.
</details>
