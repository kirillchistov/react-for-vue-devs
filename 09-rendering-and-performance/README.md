# Модуль 09. Рендеринг, сверка и оптимизации

## Смена со старшиной Рендер

«Засекаю время каждой операции на станции. Не потому что мне жалко процессорных тактов — потому что вижу, когда кто-то переделывает уже сделанную работу, и это раздражает больше, чем медленный компонент сам по себе.

На Proxy Drive движок сам решал, что перерисовывать: система слежения видела, какое реактивное значение прочитал компонент, и обновляла ровно те панели, которые от него зависят. Здесь, на React Core, вы уже знаете: обновление состояния перерисовывает компонент и *весь* его поддерево — без автослежения за тем, что именно из пропсов реально использовалось. По умолчанию это нормально и почти всегда достаточно быстро. Сегодняшняя смена — про те редкие случаи, когда «достаточно быстро» перестаёт быть достаточным, и что с этим делать точечно, а не тотально».

## Идея модуля: реконсиляция — не то же самое, что реактивность Vue

### Как React решает, что перерисовать

В Vue компилятор шаблонов и система реактивности вместе знают: `<PanelA :value="oxygen" />` перечитывается только если изменился именно `oxygen`. Это выведено из зависимостей автоматически.

В React при изменении состояния React **вызывает функцию компонента заново** — и по умолчанию так же поступает со всеми его дочерними компонентами, вне зависимости от того, изменились ли их пропсы. Затем React сравнивает получившееся дерево с предыдущим (это называется *сверка*, reconciliation) и обновляет в реальном DOM только то, что физически отличается. Два разных шага:

1. **Рендер** — вызов функций компонентов, построение нового виртуального дерева. Происходит для всего поддерева по умолчанию.
2. **Коммит** — сравнение с предыдущим деревом и точечное обновление настоящего DOM. Уже само по себе быстрое — React не трогает DOM-узлы, которые не изменились.

Шаг 1 — то место, где может быть лишняя работа: вызвать функцию компонента, который ничего не показывает по-другому, — это впустую потраченное время, даже если шаг 2 в итоге ничего не меняет в DOM. Именно этот лишний шаг 1 и умеет пропускать `memo`.

### `memo`: «не вызывай меня заново, если пропсы те же»

```jsx
const IngredientCard = memo(function IngredientCard({ ingredient, onAdd }) {
  // ...
});
```

`memo` сравнивает новые пропсы со старыми (по умолчанию — поверхностно, `===` для каждого поля) и, если всё совпало, вообще не вызывает функцию компонента заново — React переиспользует прошлый результат рендера целиком.

Ключевая ловушка: **функции и объекты, созданные заново при каждом рендере родителя, никогда не проходят `===`-сравнение**, даже если делают одно и то же:

```jsx
// Каждый рендер HomePage создаёт новую функцию — memo это видит
// как «пропс изменился» и перерисовывает IngredientCard всё равно.
<IngredientCard onAdd={(ingredient) => dispatch(addItem(ingredient))} />
```

### `useCallback`: чтобы функция осталась той же самой между рендерами

```jsx
const handleAdd = useCallback((ingredient) => dispatch(addItem(ingredient)), [dispatch]);
```

`useCallback` возвращает ту же самую ссылку на функцию между рендерами, пока не изменится что-то из списка зависимостей. Теперь `memo(IngredientCard)` видит один и тот же `onAdd` при каждом рендере родителя — и пропускает перерисовку, если `ingredient` тоже не изменился.

Важно: `memo` и `useCallback` — это пара. `useCallback` без `memo` на потребителе ничего не даёт (нечему сравнивать пропсы). `memo` без `useCallback` у родителя, который передаёт инлайн-функцию, тоже ничего не даёт (пропсы всё равно каждый раз новые).

## Задача

Стабилизировать перерисовку карточек ингредиентов: добавление одного ингредиента в конструктор не должно перерисовывать карточки остальных.

```
exercise/src/
  components/ingredient-card/ingredient-card.tsx   TODO — оберните в memo(...)
  pages/home.tsx                                     TODO — оберните onAdd в useCallback
```

Оба TODO нужны одновременно — по отдельности каждый бессилен (см. теорию выше). Тест `ingredient-card.test.tsx` проверяет идею в изоляции (стабильный колбэк против нестабильного), `home.test.tsx` — то же самое в реальном дереве страницы.

## Инспектор Тай уже здесь

```ts
export const IngredientCard = memo(function IngredientCard({ ingredient, onAdd }: IngredientCardProps) {
  // ...
});
```

`memo` — дженерик-функция: `memo<P>(Component: FunctionComponent<P>): FunctionComponent<P>`. Типы пропсов у обёрнутого компонента остаются ровно те же, что у исходного — TypeScript выводит `P` из переданной функции автоматически, писать `memo<IngredientCardProps>(...)` руками не нужно. Ровно та же идея вывода типа, что вы видели у `Select<TItem>` в модуле 08.

## Проверь себя

<details>
<summary>1. В чём разница между «рендером» и «коммитом» в терминологии React?</summary>

Рендер — это вызов функций компонентов и построение нового виртуального дерева; по умолчанию происходит для всего поддерева при любом изменении состояния выше. Коммит — сравнение (сверка) этого нового дерева со старым и точечное применение только реальных отличий к настоящему DOM. Лишняя работа, которую устраняет `memo`, — это лишние вызовы функций на шаге рендера, а не лишние изменения DOM на шаге коммита (DOM React и так трогает точечно).
</details>

<details>
<summary>2. Компонент обёрнут в <code>memo(...)</code>, но всё равно перерисовывается на каждый рендер родителя. Родитель передаёт <code>onClick={() => doSomething(id)}</code> прямо в JSX. Что не так?</summary>

Стрелочная функция `() => doSomething(id)` создаётся заново при каждом вызове функции родителя — это новый объект-функция в памяти, даже если делает то же самое. `memo` сравнивает пропсы через `===`, а два разных объекта-функции никогда не равны через `===`, сколько бы одинаковый код они ни содержали. Нужен `useCallback` в родителе, чтобы между рендерами передавалась одна и та же ссылка.
</details>

<details>
<summary>3. Если обернуть <code>IngredientCard</code> в <code>memo</code>, но не трогать <code>HomePage</code>, что-нибудь изменится?</summary>

Скорее всего нет, если `HomePage` передаёт `onAdd` инлайн-функцией на каждый рендер (как было до TODO этого модуля) — `memo` увидит новый `onAdd` при каждом сравнении и всё равно перерисует компонент. `memo` без стабильных пропсов у источника — красная лампочка без проводки: технически на месте, но не работает.
</details>

<details>
<summary>4. Нужно ли оборачивать в <code>memo</code> вообще каждый компонент в приложении «на всякий случай»?</summary>

Нет. У `memo` есть собственная цена — сравнение пропсов на каждый рендер родителя, пусть и дешёвое, но не бесплатное. Для компонента, который и так рендерится редко или дёшево (простой текст, маленькая разметка), сравнение пропсов может стоить дороже, чем просто перерисовать его заново. `memo` имеет смысл там, где: (а) компонент рендерится часто из-за родителя, у которого меняется несвязанное состояние, и (б) сам рендер компонента при этом заметно недешёвый — как список карточек ингредиентов, который может быть длинным.
</details>

<details>
<summary>5. В дополнительном примере при выключенном чекбоксе «Стабилизировать пропсы» счётчик рендеров панели растёт каждую секунду сам по себе. Откуда эта перерисовка, если <code>value</code> панели не менялся?</summary>

Часы (`tick`) тикают в `App` через `setInterval`, обновляя состояние `App` каждую секунду — это перерисовывает `App` целиком. При этом `App` создаёт `onRecalibrate` инлайн-функцией внутри своего тела, то есть новую ссылку на каждый такой рендер. Хотя `value` панели действительно не менялся, `memo` видит, что *другой* пропс (`onRecalibrate`) стал другим объектом, и перерисовывает панель целиком — ровно то же самое, что происходило с `IngredientCard` до TODO этого модуля.
</details>
