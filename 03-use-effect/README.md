# Модуль 03. `useEffect` — не `watch`

## Смена с лейтенант-коммандером Эффектом

«С этой смены — станция переходит на настоящую связь с внешним миром. Ингредиенты больше не лежат готовым массивом в коде — их нужно запросить у склада, подождать ответ, и быть готовым, что склад не откликнется. А ещё добавляем поиск по каталогу, потому что пролистывать девять ингредиентов вручную — такое.

Здесь и начнётся самая частая путаница у инженеров с вашим прошлым. На Proxy Drive у вас был `watch` — он реагирует, когда меняется конкретное значение, и мыслит в терминах "что изменилось". `useEffect` не такой. Он не реагирует на изменения — он синхронизирует компонент с чем-то внешним (сервер, таймер, подписка) и перезапускается по списку зависимостей, который вы указываете сами. Разница кажется незначительной, пока вы не напишете код, который либо крутится в бесконечном цикле, либо продолжает делать вид, что ничего не изменилось».

## Идея модуля: `useEffect` — это не подписка на изменение

В Vue `watch(source, callback)` и `watchEffect(callback)` работают через автослежение: Proxy знает, какие реактивные значения читает ваш callback, и запускает его заново именно тогда, когда одно из этих значений меняется. Модель — «отреагировать на изменение».

`useEffect(callback, deps)` устроен иначе. React не отслеживает, что вы читаете внутри `callback` — он просто сравнивает массив `deps` с предыдущим рендером и решает, перезапускать эффект или нет. Модель — «синхронизировать что-то с текущими значениями пропсов/state», а не «отреагировать на изменение конкретного значения». Именно поэтому пустой массив `[]` означает «выполнить один раз при монтировании и синхронизировать с внешним миром один раз», а не «наблюдать за пустым списком значений».

```js
// Vue: watch реагирует на конкретный источник
watch(searchQuery, (newQuery) => {
  fetchResults(newQuery);
});
```

```jsx
// React: useEffect синхронизирует эффект с текущим searchQuery
useEffect(() => {
  fetchResults(searchQuery);
}, [searchQuery]); // перезапустится, когда searchQuery станет другим
```

Выглядит похоже на этом примере — расхождение начинается с **cleanup** и с **устаревших замыканий (stale closures)**, которых у `watch` в такой форме просто нет.

### Cleanup — это не «дополнительная опция», это половина контракта

Функция, которую вы возвращаете из `useEffect`, выполняется перед каждым следующим запуском эффекта и при размонтировании компонента. Без неё побочные эффекты (открытые соединения, таймеры, устаревшие запросы) переживают компонент, который их создал.

```jsx
useEffect(() => {
  const timeoutId = setTimeout(() => setDebouncedValue(value), 300);
  return () => clearTimeout(timeoutId); // отменяет предыдущий таймер
}, [value]);
```

### Устаревшее замыкание — баг, у которого нет прямого аналога в Vue

Функция внутри `useEffect` — обычное JS-замыкание. Она захватывает те значения переменных, что были на момент **этого конкретного рендера**, и если эффект не перезапускается (пустой массив зависимостей или неполный список), эти значения замораживаются — даже если сам компонент продолжает перерисовываться по другим причинам.

```jsx
useEffect(() => {
  const id = setInterval(() => {
    setCount(count + 1); // count всегда 0 — замыкание из первого рендера
  }, 1000);
  return () => clearInterval(id);
}, []); // count не в зависимостях — эффект не «видит» новых значений count
```

В Vue такого класса багов почти нет: `watch`/`watchEffect` всегда читают текущее значение `ref`/`reactive` через `.value`, а не захватывают его копию на момент какого-то рендера.

## Инспектор Тай уже здесь

`fetchIngredients` типизирован через возвращаемый `Promise`:

```ts
export function fetchIngredients(): Promise<TIngredient[]> {
  return new Promise((resolve) => {
    setTimeout(() => resolve(ingredients), 50);
  });
}
```

Обратите внимание на `TCatalogStatus`, который расширился до `'loading' | 'ready' | 'error'`. В Vue вы бы, скорее всего, держали отдельно `isLoading`/`error` как булевы флаги (и рисковали получить состояние `isLoading: true, error: 'что-то'` одновременно — противоречивое и вроде бы невозможное, но ничем не запрещённое). Объединение в один union-тип статуса делает недопустимые комбинации невозможными на уровне типов — не два независимых флага, а одно значение из трёх вариантов.

## Проверь себя

<details>
<summary>1. Эффект зависит от <code>[]</code> и внутри читает <code>props.userId</code>. Пользователь сменился, <code>userId</code> в пропсах обновился. Увидит ли эффект новое значение?</summary>

Нет. Пустой массив зависимостей означает «запустить один раз при монтировании и никогда не перезапускать». Замыкание внутри эффекта захватило `userId` таким, каким он был при первом рендере, и будет использовать именно это значение при каждом вызове (например, в колбэке таймера), даже если реальный `props.userId` давно другой.
</details>

<details>
<summary>2. Зачем в эффекте загрузки нужен флаг <code>ignore</code>, если промис и так рано или поздно завершится?</summary>

Промис действительно завершится, но компонент может успеть размонтироваться до этого момента (например, пользователь ушёл на другую страницу). Вызов `setState` на уже размонтированном компоненте — предупреждение (а в некоторых версиях React — ошибка) и потенциальная утечка. Флаг `ignore`, выставляемый в `true` в функции очистки, не отменяет сам запрос, но не даёт его результату повлиять на состояние компонента, которого уже нет.
</details>

<details>
<summary>3. В чём разница между тем, что происходит при смене зависимости, и тем, что происходит при размонтировании?</summary>

Функция очистки вызывается в обоих случаях, но при смене зависимости сразу вслед за ней запускается новая версия эффекта — то есть смена зависимости выглядит как «очистить старое, затем настроить новое». При размонтировании функция очистки вызывается, а нового запуска уже не будет.
</details>

<details>
<summary>4. Почему debounce реализуют через <code>setTimeout</code> внутри эффекта с зависимостью от значения инпута, а не через таймер снаружи React?</summary>

Потому что при каждом изменении `searchInput` функция очистки эффекта (вызываемая перед следующим запуском) отменяет предыдущий, ещё не сработавший таймер через `clearTimeout`. Так гарантируется, что «выстрелит» только самый последний таймер — если бы таймер жил вне React и не отменялся, каждое нажатие клавиши планировало бы новое обновление поверх старых.
</details>

<details>
<summary>5. Что бы изменилось в поведении, если бы в зависимостях эффекта загрузки был указан <code>[fetchIngredients]</code> вместо <code>[]</code>?</summary>

Формально ничего не изменилось бы прямо сейчас — `fetchIngredients` объявлена вне компонента и ссылка на неё не меняется между рендерами, так что эффект всё равно выполнится один раз. Но это создаёт ложное впечатление, что перезапуск зависит от функции, и если в будущем `fetchIngredients` станет пропом или результатом другого хука с нестабильной ссылкой на каждый рендер — эффект начнёт перезапускаться на каждом рендере незаметно для автора изменения.
</details>
