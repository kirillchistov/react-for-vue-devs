# Модуль 05. Глобальное состояние: Context и Redux Toolkit

## Смена с дроидом-квартирмейстером QM-9 «Редакс»

«ЗАПРОС ПРИНЯТ. Раньше состояние конструктора жило внутри одного хука, и добраться до него могли только `App` и его прямые дети. Теперь на станции появляется новая панель — счётчик в шапке, — а шапка не является предком ни каталога, ни конструктора. Тащить `count` через десяток промежуточных компонентов пропами я, квартирмейстер, не одобряю.

На Vuex или Pinia вы бы, скорее всего, просто мутировали `state.items` напрямую. ЗДЕСЬ ТАК НЕЛЬЗЯ. Единственный способ что-то изменить в моём журнале склада — подать официальную заявку (экшен). Журнал сам решит, как её обработать».

## Идея модуля: почему вообще не пропсы

До этого момента состояние конструктора жило в `useConstructor` — обычном хуке, локальном для `App`. Это отлично работало, пока состояние было нужно только внутри одного поддерева компонентов. Как только появляется компонент из **другой** ветки дерева (шапка), у локального состояния и prop drilling заканчиваются варианты: либо поднимать состояние ещё выше и тащить его пропами через всё дерево, либо дать компонентам читать его напрямую.

### Context — читать напрямую, без прокидывания

```jsx
// Кто угодно внутри Provider может прочитать значение без пропов
const ThemeContext = createContext(null);

function StatusCard() {
  const { theme } = useContext(ThemeContext); // не родитель и не потомок ThemeToggle
}
```

Это уже разбиралось в модуле 00 как аналог `provide`/`inject`. Ловушка та же: любое изменение значения Context перерисовывает **все** компоненты, которые его читают, — Context не различает, какое именно поле объекта вас интересует.

### Redux Toolkit — то же самое, но с чётким протоколом изменений

Context прекрасно решает задачу «прочитать без пропов», но ничего не говорит о том, *как* это значение меняется — это всё ещё произвольный `setState` где угодно. Redux (в том числе Toolkit) добавляет протокол: единственный способ поменять state — сообщить об этом через экшен, а решает, что с этим экшеном делать, — reducer.

```js
// Vuex/Pinia: mutation — это и есть API, никакого протокола не нужно
state.items.push(newItem);
```

```js
// Redux Toolkit: state меняется только через reducer, вызванный экшеном
dispatch(addItem(newItem));
// reducer сам решает, как это отразится на состоянии:
addItem: (state, action) => {
  state.items.push({ uid: crypto.randomUUID(), ingredient: action.payload });
}
```

Обратите внимание: **внутри** reducer `createSlice` мутирующий на вид код (`state.items.push(...)`) — это нормально и единственное официально разрешённое исключение из иммутабельного правила Redux, потому что Immer (библиотека под капотом Toolkit) сама строит из этого новое неизменяемое состояние. Мутировать `items` где угодно **вне** reducer'а (например, прямо в компоненте) по-прежнему нельзя — это как раз то же самое правило из модуля 02, никуда не делось.

## Задача

Перевести состояние конструктора с локального хука на Redux Toolkit slice, добавить в шапку счётчик, который читает это состояние напрямую.

```
exercise/src/
  services/slices/burger-slice.ts          TODO — reducers addItem/removeItem
  components/app/app.tsx                    TODO — handleAdd/handleRemove через dispatch
  components/app-header/app-header.tsx      TODO — count через useAppSelector
```

`services/store.ts`, `hooks/use-app-dispatch.ts` и `hooks/use-app-selector.ts` уже готовы — это стандартная обвязка, которую в реальных проектах пишут один раз и не трогают. Загрузка каталога (`useIngredients`) осталась локальным хуком: она не нужна компонентам за пределами каталога, так что превращать её в Redux незачем — не всё состояние обязано быть глобальным.

## Инспектор Тай уже здесь

Типизированные хуки — стандартный паттерн Redux Toolkit + TypeScript:

```ts
export const useAppDispatch: () => AppDispatch = useDispatch;
export const useAppSelector: TypedUseSelectorHook<RootState> = useSelector;
```

Без этого `useSelector`/`useDispatch` из `react-redux` не знают формы вашего конкретного store и типизируют всё как `unknown`. `RootState` и `AppDispatch` выводятся автоматически из самого `store` (`ReturnType<typeof store.getState>`), так что при добавлении нового slice эти типы обновятся сами — переписывать `use-app-selector.ts` не придётся никогда.

## Проверь себя

<details>
<summary>1. Компонент читает <code>useContext(ThemeContext)</code>, но использует только поле <code>theme</code>, а сам Context хранит <code>{ theme, toggleTheme }</code>. Перерисуется ли этот компонент, если поменяется только <code>toggleTheme</code> (гипотетически)?</summary>

Да. Context не отслеживает, какое конкретно поле объекта-значения читает компонент, — при любом изменении значения, переданного в `Provider`, перерисовываются все компоненты, вызывающие `useContext` для этого контекста, независимо от того, какая часть объекта им реально нужна.
</details>

<details>
<summary>2. Почему <code>state.items.push(...)</code> внутри reducer'а Redux Toolkit — это нормально, а <code>burgerItems.push(...)</code> в компоненте (модуль 02) — нет?</summary>

`createSlice` оборачивает каждый reducer в Immer, которая перехватывает «мутирующий» код и строит по нему новое неизменяемое состояние — это единственное официально разрешённое исключение. В обычном компоненте (или где угодно вне reducer'а) такой обёртки нет: мутация массива не меняет ссылку на него, и React/Redux не видят повода ни на что реагировать.
</details>

<details>
<summary>3. Зачем нужны <code>useAppDispatch</code>/<code>useAppSelector</code>, если можно использовать <code>useDispatch</code>/<code>useSelector</code> из <code>react-redux</code> напрямую?</summary>

Обычные `useDispatch`/`useSelector` ничего не знают о форме конкретно вашего store — `useSelector` будет типизировать `state` как `unknown`, и придётся вручную указывать тип в каждом месте вызова. Типизированные обёртки один раз связывают хуки с `RootState`/`AppDispatch` конкретного проекта — дальше автодополнение и проверка типов работают из коробки везде.
</details>

<details>
<summary>4. Загрузка каталога ингредиентов (<code>useIngredients</code>) осталась локальным хуком, а не переехала в Redux вместе с конструктором. Это недосмотр?</summary>

Нет, осознанное решение по масштабу. Redux (или Context) нужен, когда состояние требуется компонентам из разных, не связанных предок-потомок веток дерева — как случилось со счётчиком в шапке. Список ингредиентов каталога нужен только внутри самого каталога; превращать его в глобальное состояние добавило бы сложности без выгоды.
</details>

<details>
<summary>5. Тест вызывает <code>burgerReducer(undefined, addItem(ingredient))</code> напрямую, без всякого <code>store</code> или <code>Provider</code>. Как это вообще возможно?</summary>

Reducer, экспортированный из `createSlice`, — это просто чистая функция `(state, action) => newState`, а `store` — лишь механизм, который вызывает эту функцию и хранит результат. Так как reducer чистый и не имеет скрытых зависимостей от React, его можно тестировать напрямую, без рендеринга компонентов и без обёртки в `Provider` — это самый быстрый и надёжный способ проверить бизнес-логику Redux.
</details>
