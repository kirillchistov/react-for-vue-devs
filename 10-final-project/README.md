## Идея модуля: три темы, одна цепочка

### 1. Ещё один слайс — тот же Redux Toolkit, что и в модуле 05

`auth-slice` устроен ровно как `burger-slice`: `createSlice`, два редьюсера, никакого нового API.

```ts
export const authSlice = createSlice({
  name: 'auth',
  initialState: { user: null as TAuthUser | null },
  reducers: {
    login: (state, action: PayloadAction<TAuthUser>) => {
      state.user = action.payload;
    },
    logout: (state) => {
      state.user = null;
    },
  },
});
```

Разница с `burgerSlice` — не в механике, а в том, что теперь в `store.ts` два независимых слайса, и `AppHeader` читает данные из обоих одним и тем же `useAppSelector`, ничего не зная о том, как каждый из них устроен внутри.

### 2. Защищённый маршрут — `<Navigate>` из модуля 06, только на выходе, а не в модалке

В модуле 06 `navigate(-1)` закрывал модалку. Здесь та же функция семьи `react-router-dom` — компонент `<Navigate>` — решает другую задачу: не дать отрендериться странице, на которую пользователю сейчас нельзя.

```jsx
function ProtectedRoute({ children }) {
  const user = useAppSelector((state) => state.auth.user);

  if (user === null) {
    return <Navigate to="/login" replace />;
  }

  return children;
}
```

`ProtectedRoute` — не отдельный маршрут, а обёртка вокруг `element` уже существующего `<Route>`:

```jsx
<Route
  path="/profile"
  element={
    <ProtectedRoute>
      <ProfilePage />
    </ProtectedRoute>
  }
/>
```

Обёртка проверяет условие один раз и либо рендерит `children`, либо подменяет их на `<Navigate>` — тот же React-принцип «компонент возвращает разные деревья в зависимости от пропсов/состояния», что вы видели с первого модуля, просто применённый к решению «пускать или не пускать».

`replace` — важная деталь: без него в истории браузера остаётся запись о `/profile`, и кнопка «назад» после успешного логина вернула бы пользователя обратно на страницу, на которую он и так не должен был попадать. `replace` заменяет текущую запись истории вместо добавления новой.

### 3. Форма входа — тот же паттерн, что и в модуле 07, только результат идёт в Redux, а не в локальный `useState`

```jsx
const handleSubmit = (values) => {
  dispatch(login({ email: values.email }));
  navigate('/profile');
};
```

`LoginForm` из модуля 07 не изменилась ни на строку — она как принимала `onSubmit`, так и принимает. Изменилось только то, что происходит *после* валидной отправки: раньше — `setLoggedInEmail` в локальном состоянии страницы, теперь — `dispatch` в общий стор плюс `navigate` на защищённую страницу. Компонент, который ничего не знает о Redux и роутинге, остаётся переиспользуемым в обоих случаях — ровно то, ради чего в модуле 04 разделяли «что показать» и «что сделать при сабмите».

## Задача

Дособрать цепочку «вход → защищённая страница → выход».

```
exercise/src/
  services/slices/auth-slice.ts                     TODO — редьюсеры login/logout
  components/protected-route/protected-route.tsx     TODO — редирект на /login
  pages/login.tsx                                     TODO — dispatch(login(...)) + переход на /profile
  pages/profile.tsx                                   TODO — dispatch(logout()) + переход на /
```

Все четыре TODO нужны одновременно — это одна цепочка, а не четыре независимые задачи: без `login`/`logout` в слайсе нечего диспатчить; без редиректа в `ProtectedRoute` защиты нет вообще; без диспатча на страницах слайс никогда не заполнится и не очистится.

## Проверь себя

<details>
<summary>1. Чем <code>ProtectedRoute</code> отличается от обычного компонента, который вы писали раньше?</summary>

Ничем принципиально — это обычный компонент, который получает `children` пропом и решает, отрендерить их как есть или подменить на `<Navigate>`. Никакой особой магии react-router здесь нет: `<Route element={...}>` просто рендерит то дерево, которое вы туда передали, а `ProtectedRoute` — часть этого дерева, применяющая обычную условную логику рендера, знакомую с модуля 01.
</details>

<details>
<summary>2. Зачем в <code>&lt;Navigate to="/login" replace /&gt;</code> нужен <code>replace</code>?</summary>

Без `replace` переход на `/login` добавляет новую запись в историю браузера поверх записи о `/profile`, на которую пользователь и так не должен был попасть. После успешного входа кнопка «назад» вернула бы его снова на `/profile` — то есть на мгновение показала бы защищённую страницу, а затем `ProtectedRoute` немедленно увёл бы его обратно. `replace` заменяет запись `/profile` в истории на `/login`, и такого прыжка не происходит.
</details>

<details>
<summary>3. Почему <code>LoginForm</code> из модуля 07 не пришлось менять ни на строку в этом модуле?</summary>

`LoginForm` с самого начала не знала ничего ни о Redux, ни о роутинге — она принимала `onSubmit` и вызывала его с валидными `{ email, password }`, а что делать с этими данными, решал вызывающий код. В модуле 07 `onSubmit` клал email в локальный `useState`; в этом модуле `onSubmit` кладёт его в Redux и переключает маршрут. Компонент формы остался прежним, потому что с самого начала не был жёстко связан с тем, куда идут его данные — тот же принцип разделения ответственности, что и у `OrderSubmit`, который тоже не знает, откуда берутся ингредиенты.
</details>

<details>
<summary>4. Если убрать <code>ProtectedRoute</code> вокруг <code>/profile</code>, но оставить <code>auth-slice</code> и логику входа/выхода нетронутыми — что сломается?</summary>

Сама авторизация продолжит работать: вход по-прежнему запишет пользователя в Redux, выход — очистит. Но `/profile` перестанет быть защищённым маршрутом — любой человек, введя адрес напрямую или кликнув по ссылке «Профиль», увидит личный кабинет, даже не логинившись (просто `user?.email` окажется `undefined`, и вместо email там будет пусто). `ProtectedRoute` — это единственное звено цепочки, которое превращает «у нас есть данные о пользователе» в «эта страница действительно недоступна без входа».
</details>

<details>
<summary>5. В дополнительном примере тот же паттерн работает без Redux — на голом <code>useState</code>. Значит ли это, что Redux вообще не нужен для авторизации?</summary>

Нет — паттерн защиты маршрута (`ProtectedRoute` + `<Navigate>`) не завязан на то, откуда берётся условие «есть доступ или нет». В маленьком примере это локальный `useState` в одном компоненте `App`, потому что состояние нужно только там. В капстоуне тот же флаг нужен одновременно `ProtectedRoute`, `AppHeader` и `LoginPage` — трём разным компонентам в разных частях дерева, — и вот для этого случая как раз пригождается общее хранилище из модуля 05. Выбор между `useState` и Redux — вопрос того, сколько разных мест в дереве должны знать об одном и том же состоянии, а не про авторизацию саму по себе.
</details>
