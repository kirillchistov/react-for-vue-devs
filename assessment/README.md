# Входная диагностика

Перед тем как выдать вам пропуск на станцию, капитан Рут просит пройти короткую диагностику. Это не экзамен и не фильтр — здесь никого не разворачивают у трапа. Цель ровно одна: понять, какие смены (модули) можно пробежать по диагонали, а какие — пройти от и до, потому что именно там сидит рабочая, но неверная для React привычка.

10 вопросов, каждый привязан к конкретному модулю. У каждого варианта ответа — свой разбор, а не только у правильного: неправильный вариант тут не «минус балл», а часто ровно та ошибка, которую подсказывает Vue-рефлекс.

Ответ засчитан, только если выбраны **все** верные варианты (в паре вопросов их больше одного).

---

### 1. Мутация state
```jsx
function BurgerCart() {
  const [items, setItems] = useState(['булка']);

  const addSauce = () => {
    items.push('соус');
    setItems(items);
  };

  return (
    <div>
      <button onClick={addSauce}>Добавить соус</button>
      <p>{items.length}</p>
    </div>
  );
}
```
Что произойдёт при клике на кнопку?

- [ ] Счётчик на экране увеличится до 2. — Неверно: `items` мутируется на месте, а в `setItems` уходит та же самая ссылка на массив.
- [x] Счётчик на экране останется 1, хотя реально в массиве уже два элемента. — Верно: React сравнивает ссылку на state между рендерами. Ссылка не поменялась — повода перерисовывать компонент нет, даже если содержимое массива изменилось.
- [ ] React выбросит ошибку, потому что state мутировать нельзя. — Неверно: ошибки не будет, будет тихий баг — самый неприятный вид багов.
- [ ] React сам предупредит в консоли о мутации state. — Неверно: `useState` не отслеживает мутации и ничего не логирует.

→ Промахнулись — весь модуль **02-state-and-reactivity** ваш, без сокращений.

### 2. `useEffect` и устаревшее замыкание
```jsx
function OrderCounter() {
  const [count, setCount] = useState(0);

  useEffect(() => {
    const id = setInterval(() => {
      setCount(count + 1);
    }, 1000);
    return () => clearInterval(id);
  }, []);

  return <p>{count}</p>;
}
```
Что покажет счётчик через 5 секунд?

- [ ] 5, 6, 7... — счётчик растёт каждую секунду. — Неверно.
- [x] 1 — и дальше не меняется. — Верно: замыкание внутри эффекта захватило `count = 0` из первого рендера. Эффект не перезапускается (`deps = []`), поэтому каждую секунду выполняется один и тот же `setCount(0 + 1)`.
- [ ] Приложение упадёт с ошибкой "count is not defined". — Неверно.
- [ ] Будет то же поведение, что и с `watchEffect` в Vue на реактивном `count`. — Неверно: `watchEffect` в Vue отслеживает изменение `count.value` через Proxy и пересчитывается автоматически при каждом изменении — в React такого автослежения нет в принципе.

→ Промахнулись — это ядро модуля **03-use-effect**, читайте про stale closures отдельно.

### 3. `useRef` не перерисовывает компонент
```jsx
function Stopwatch() {
  const secondsRef = useRef(0);

  const tick = () => {
    secondsRef.current += 1;
  };

  return (
    <div>
      <button onClick={tick}>Тик</button>
      <p>Секунды: {secondsRef.current}</p>
    </div>
  );
}
```
Что произойдёт при нескольких кликах на кнопку?

- [ ] Число на экране будет расти с каждым кликом. — Неверно.
- [x] `secondsRef.current` реально увеличивается в памяти, но текст на экране не обновится, пока что-то ещё не вызовет перерисовку компонента. — Верно: изменение `.current` не планирует рендер — это специально спроектированное поведение `useRef`.
- [ ] Будет ошибка — `useRef` нельзя мутировать в обработчике события. — Неверно, мутировать можно и нужно.
- [ ] То же самое, что с `ref()` в Vue — компонент перерисуется сам. — Неверно и это ключевая ловушка: одно и то же слово «ref» в Vue и в React означает два разных механизма.

→ Промахнулись — обратите внимание на разницу `useState`/`useRef` в модуле **02-state-and-reactivity**.

### 4. Условный вызов хука
```jsx
function IngredientCard({ ingredient, isSelected }) {
  if (isSelected) {
    const [highlighted, setHighlighted] = useState(true);
  }

  return <div>{ingredient.name}</div>;
}
```
Что не так с этим компонентом?

- [ ] Ничего, это нормальный способ подключать состояние по условию. — Неверно.
- [x] `useState` вызван внутри условия — порядок вызовов хуков между рендерами может не совпасть, а React связывает состояние хука именно с порядковым номером вызова. — Верно.
- [ ] Проблема в том, что `highlighted` нигде не используется. — Неверно: это отдельная (валидная) претензия линтера, но не причина, по которой ломается состояние.
- [ ] В Vue composables можно вызывать условно, значит и в React так можно. — Неверно: у composables нет такого жёсткого правила о порядке вызова, у hooks — есть, и это осознанное архитектурное решение.

→ Промахнулись — «Правила хуков» разбираются в модуле **04-composition**.

### 5. Мутация state внутри Redux Toolkit reducer
```js
const cartSlice = createSlice({
  name: 'cart',
  initialState: { items: [] },
  reducers: {
    addItem(state, action) {
      state.items.push(action.payload);
    },
  },
});
```
Это Redux Toolkit reducer, и он «мутирует» `state.items` напрямую через `push`. Это ошибка?

- [ ] Да, любая мутация state в Redux — гарантированный баг. — Неверно в данном контексте.
- [x] Нет: `createSlice` оборачивает reducer в Immer, которая по описанным «мутациям» строит новое неизменяемое состояние под капотом. Это единственное официально разрешённое исключение из иммутабельного правила Redux. — Верно.
- [ ] Нет, потому что Redux Toolkit работает так же, как Pinia — state там в принципе мутируется без всякого Immer. — Неверно: результат снаружи выглядит похоже, но механизм совершенно другой, и вне reducer'а (например, прямо в компоненте) так мутировать state по-прежнему нельзя.
- [ ] Это сработает только в dev-режиме, в проде поломается. — Неверно.

→ Промахнулись — нюансы Redux Toolkit и Immer разберём в модуле **05-global-state**.

### 6. Модальный маршрут
Вы открываете детали ингредиента через модалку: в адресной строке появляется `/ingredients/123`, а фон остаётся конструктором бургера. Если обновить страницу (F5), находясь на этом URL, при правильной реализации должно открыться...

- [ ] Пустая страница — модалка это состояние компонента, а не отдельный маршрут. — Неверно, и это как раз то, что не должно происходить при правильной реализации.
- [x] Полноценная страница с деталями ингредиента, уже без фона-конструктора — потому что это самостоятельный маршрут, который лишь *опционально* рендерится поверх фона, когда переход был внутренним. — Верно.
- [ ] Ошибка 404 — React Router не поддерживает такие ссылки. — Неверно.
- [ ] Ровно то же самое модальное окно без фона — как named views в Vue Router. — Неверно: похожая идея, но реализуется другим механизмом (background location), не прямым аналогом named views.

→ Промахнулись — модальные маршруты разбираются в модуле **06-routing**.

### 7. Контролируемый инпут без `onChange`
```jsx
function LoginForm() {
  const [email, setEmail] = useState('');

  return <input value={email} />;
}
```
Что произойдёт, если попробовать печатать в этом поле?

- [x] Поле окажется заблокированным для ввода, а React в dev-режиме выведет в консоль предупреждение про `value` без `onChange`. — Верно.
- [ ] Поле будет работать как обычное неконтролируемое поле ввода. — Неверно.
- [ ] React выбросит ошибку, и приложение упадёт. — Неверно, это предупреждение, а не ошибка выполнения.
- [ ] То же самое, что `v-model` без обработчика в Vue. — Неверно: у `v-model` нет сценария «без обработчика» — он сам генерирует и значение, и обработчик.

→ Промахнулись — контролируемые формы разбираются в модуле **07-forms-and-data**.

### 8. Типизация `children`
```tsx
function Modal({ children }) {
  return <div className="modal">{children}</div>;
}
```
Как правильно типизировать пропсы этого компонента?

- [ ] `{ children: string }` — Неверно: `children` почти никогда не просто строка.
- [x] `{ children: React.ReactNode }` — Верно: `ReactNode` покрывает всё, что можно передать в JSX — элементы, строки, массивы, `null`, фрагменты.
- [ ] `{ children: JSX.Element }` — Неверно: это только один-единственный React-элемент, без строк, массивов и `null`.
- [ ] Типизировать не нужно — как с `defineProps`, тип выведется сам. — Неверно: в React нет компиляторных макросов, только обычные TS-типы, которые нужно написать явно.

→ Промахнулись — типизация пропсов и `children` — в модуле **08-typescript**.

### 9. `key={index}` на изменяемом списке
```jsx
{items.map((item, index) => (
  <IngredientRow key={index} ingredient={item} onRemove={() => remove(index)} />
))}
```
Пользователь удаляет ингредиент из середины списка. Что может пойти не так?

- [ ] Ничего, `key` есть — этого достаточно. — Неверно: `key` есть, но он завязан на позицию, а не на сам элемент.
- [x] React может «перепутать» локальное состояние строк между собой (например, открытое раскрытие или чекбокс), потому что после удаления элемента все ключи после него сдвигаются. — Верно.
- [ ] Список отсортируется в обратном порядке. — Неверно.
- [ ] Приложение упадёт с ошибкой "duplicate key". — Неверно: дубликатов ключей тут не возникает, баг куда тише.

→ Промахнулись — reconciliation и ключи разбираются в модуле **09-rendering-and-performance**.

### 10. Гранулярность Context
Вы положили в один Context объект `{ user, theme, cartCount }` и меняете `cartCount` при каждом добавлении ингредиента. Что произойдёт с компонентом, который читает из этого контекста только `theme`?

- [ ] Он не перерисуется, потому что `theme` не менялся. — Неверно.
- [x] Он тоже перерисуется: React не различает, какое именно поле объекта использует конкретный потребитель — при любом изменении значения провайдера перерисовываются все компоненты, вызывающие `useContext` для этого контекста. — Верно.
- [ ] Он перерисуется, только если явно обернуть чтение в `useMemo`. — Неверно: `useMemo` тут ничего не меняет без дополнительного разделения контекстов.
- [ ] Так же, как с `provide`/`inject` в Vue — там `inject` тоже перерисует всех подряд. — Неверно: в Vue шаблон обновит только те узлы, которые реально читают изменившееся реактивное свойство.

→ Промахнулись — гранулярность Context и когда переходить на Redux — модуль **05-global-state**.
