# Модуль 01. Компоненты и JSX без магии шаблонов

## Смена с капитаном Рутом

«Первая смена простая: разложить ингредиенты по полкам. У вас уже есть склад — вот он, в `data/ingredients.ts`, ничего готовить не нужно. Ваша задача — построить витрину, на которой всё это видно: карточки, категории, и что показать, пока склад ещё грузится или если он вдруг пуст.

Если бы вы делали это на Proxy Drive, вы бы написали шаблон с `v-for` и `v-if`. Тут шаблонов нет вообще — есть функция, которая возвращает разметку. Прежде чем трогать код, разберитесь, почему это не одно и то же».

## Идея модуля: JSX — это не шаблон

В Vue у компонента отдельный язык шаблона, который Vue-компилятор превращает в render-функцию за вас. Директивы (`v-if`, `v-for`, `v-bind`, `v-on`) — это специальный синтаксис именно этого языка.

В React разметка — часть той же функции, что и вся остальная логика. JSX — это просто синтаксический сахар над обычными вызовами функций (`React.createElement(...)`), а не отдельный язык. Из этого следует главное: **внутри `{}` в JSX — обычное JS-выражение**, а не что-то особое.

Один и тот же компонент на Vue и на React:

```vue
<!-- Vue: шаблон + директивы -->
<template>
  <article class="ingredient-card">
    <span>{{ ingredient.image }}</span>
    <p>{{ ingredient.name }}</p>
    <p>{{ ingredient.price }} ₽</p>
  </article>
</template>
```

```jsx
// React: JSX — просто JS-выражение внутри функции
function IngredientCard({ ingredient }) {
  return (
    <article className="ingredient-card">
      <span>{ingredient.image}</span>
      <p>{ingredient.name}</p>
      <p>{ingredient.price} ₽</p>
    </article>
  );
}
```

Заметьте: `class` стал `className` (`class` — зарезервированное слово в JS, поэтому атрибут переименовали), а `{{ }}` стало `{ }` — потому что это не специальный синтаксис интерполяции, а буквально JS-выражение, которое JSX вычисляет как есть.

То же самое с директивами:

| Что нужно | Vue | React |
|---|---|---|
| Список | `<li v-for="i in items" :key="i.id">` | `{items.map((i) => <li key={i.id}>...)}` |
| Условие | `<div v-if="cond">...</div>` | `{cond && <div>...</div>}` или ранний `return` |
| Атрибут из переменной | `:src="url"` | `src={url}` |

Директив в JSX нет вообще — есть `.map()`, тернарник, `&&`, ранний `return` и всё остальное, что вы и так знаете из обычного JS.

## Инспектор Тай уже здесь

TypeScript в этом курсе — с первого модуля, но не как отдельная тема, а как то, что уже написано за вас, пока вы разбираетесь с JSX. Посмотрите на `IngredientCardProps` в файле-заготовке:

```ts
interface IngredientCardProps {
  ingredient: Ingredient;
}

export function IngredientCard({ ingredient }: IngredientCardProps) { ... }
```

В Vue + `<script setup lang="ts">` вы бы написали `defineProps<{ ingredient: Ingredient }>()` — компиляторный макрос, специальный синтаксис только для этого случая. В React пропсы — это просто первый (и единственный) параметр обычной функции, и типизируется он обычным TS-типом, без всякой магии. `interface` тут ничем не отличается от типизации параметра любой другой функции, которую вы писали раньше.

В этом модуле весь тип уже дан — просто прочитайте его перед тем, как писать JSX внутри функции. С модуля 08 типы начнёте писать сами, и там уже разберём более тонкие случаи.

## Проверь себя

<details>
<summary>1. Что отрендерится, если <code>ingredient.price</code> равен 300, а код — <code>{ingredient.price > 500 && &lt;span&gt;дорого&lt;/span&gt;}</code>?</summary>

Ничего видимого. `300 > 500` — `false`, и всё выражение `false && <span>...</span>` вычисляется в `false`. React не рендерит `false`, `null`, `undefined` и `true` — они просто ничего не выводят на экран (в отличие, например, от `0`, который отрендерится как текст «0» — частая ловушка с этим же паттерном).
</details>

<details>
<summary>2. Почему нельзя написать <code>&lt;div&gt;{if (x) { return 1 }}&lt;/div&gt;</code> прямо в JSX?</summary>

Потому что внутри `{}` в JSX ожидается **выражение** (expression), а `if` — это **инструкция** (statement), она не возвращает значение. По той же причине нельзя писать туда циклы `for` или объявления переменных через `let`. Условную логику либо выносят до `return` (обычным `if` в теле функции), либо заменяют выражением — тернарником или `&&`.
</details>

<details>
<summary>3. Зачем нужен <code>key</code> в <code>.map()</code>, если список и так визуально рендерится правильно?</summary>

`key` — это не про то, что видно на экране сразу, а про то, как React сопоставляет элементы между рендерами при изменении списка (добавление, удаление, сортировка). Без стабильного `key` React может по ошибке решить, что переиспользует не тот DOM-узел и не то внутреннее состояние компонента — баг, который не виден на статичном списке, но проявляется ровно в момент, когда список меняется.
</details>

<details>
<summary>4. Чем <code>className</code> в JSX отличается от <code>class</code> в HTML/Vue-шаблоне?</summary>

Ничем, кроме имени. JSX в итоге превращается в вызовы `React.createElement`, а `class` — зарезервированное слово в JavaScript (используется для объявления классов), поэтому атрибут для CSS-класса переименовали в `className`, чтобы не конфликтовать с синтаксисом языка.
</details>

<details>
<summary>5. Компонент возвращает <code>null</code>. Что увидит пользователь на странице?</summary>

Ничего — в точке, где отрендерен этот компонент, не появится ни одного DOM-узла. `null` — официально поддерживаемое React «ничего не рендерить», в отличие от `undefined`, `false` или пустой строки, которые тоже ничего не выводят, но не считаются идиоматичным способом это сделать.
</details>
