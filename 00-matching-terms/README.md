# Модуль 0. Карта курса и шпаргалка Vue → React

Только чтение — код здесь не пишем. Задача модуля: получить общую карту курса и опорную таблицу понятий, к которой можно возвращаться в любой момент.

## Брифинг капитана Рута

«Значит, вы с Vue Nebula. Хорошая новость: вы уже знаете, что такое компонент, пропс и состояние — переучивать это не будем. Плохая новость: у вас в голове сидит Proxy Drive, который сам решает, что перерисовать. Здесь такого нет. React Core перерисовывает функцию компонента целиком, когда вы явно об этом просите, и ни секундой раньше.

Половина ошибок, которые совершают инженеры с вашим прошлым, — это не незнание React, а работающая, но неправильная здесь интуиция Vue. Наша задача на этот курс — не залить в вас React с нуля, а точечно заменить те рефлексы, которые вас подводят.

Дальше — карта смен и шпаргалка. Держите её под рукой первые несколько модулей, потом она вам не понадобится — рефлексы поменяются сами».

## Шпаргалка: Vue → React

| Vue | React-эквивалент | На что обратить внимание |
|---|---|---|
| `ref()` / `reactive()` | `useState` | В Vue мутация свойства сама обновляет экран. В React мутация state не делает ничего — нужен новый объект/массив и вызов сеттера. |
| `computed()` | `useMemo`, или просто вычисление в теле компонента | `computed` кэширует по автоматически отслеженным зависимостям. `useMemo` требует явного списка зависимостей и часто вообще не нужен — пересчёт при каждом рендере обычно дешевле, чем кажется. |
| `watch()` / `watchEffect()` | `useEffect` | Разная модель: `watch` реагирует на изменение конкретного реактивного источника, `useEffect` — это «синхронизация с внешним миром» по явному списку зависимостей, а не подписка на изменения. |
| методы (`methods`, функции в `setup()`) | обычные функции в теле компонента или в хуке | Не обязаны быть «методами объекта» — это просто функции в замыкании рендера. |
| `props` | `props` | Похоже почти один в один, но в React пропсы — единственный объект-параметр функции, без макроса `defineProps`, и его нельзя мутировать локально. |
| `emit('event')` | колбэк-проп (`onSomething`), переданный родителем | Нет встроенной шины событий — имя и поведение колбэка придумывает и передаёт сам родитель. |
| `v-model` | `value` + `onChange` (контролируемый инпут) | Двусторонней связи «из коробки» нет — её собирают руками из пары значение/обработчик. |
| `v-if` / `v-show` | тернарник, `&&`, обычное условие в JSX | Нет директив — это чистый JS-код внутри `{}`. `v-show` (просто прячет через CSS) эмулируется условным классом/стилем. |
| `v-for` | `.map()` + обязательный `key` | Забытый или «плохой» `key` (индекс массива на изменяющемся списке) — частый источник багов при пересортировке. |
| слоты (`default`, именованные) | `children`, именованные пропы с JSX внутри | У именованных слотов нет прямого аналога — их эмулируют пропами, принимающими JSX. |
| `provide` / `inject` | Context (`createContext` + `useContext`) | Context не обновляется «по частям» — любое изменение значения перерисовывает все компоненты, которые его читают. |
| миксины / composables (Composition API) | кастомные хуки | Хуки обязаны вызываться на верхнем уровне компонента — нельзя внутри условий или циклов. У composables такого жёсткого правила нет. |
| `onMounted` / `onUpdated` / `onUnmounted` | `useEffect` (+ его функция очистки) | Нет отдельного хука под каждую фазу жизненного цикла — всё моделируется одним `useEffect` и его зависимостями. |
| template ref (ref на DOM-элемент) | `useRef` | В React `useRef` используется для двух разных вещей: ссылка на DOM-узел И хранение любого мутабельного значения, которое не должно вызывать перерисовку. В Vue это два разных механизма. |
| Vuex / Pinia | Redux / Redux Toolkit / Zustand / Context | В Pinia state можно мутировать напрямую. В Redux — только через reducer и экшен, прямые мутации ломают всё. |
| Vue Router | React Router | Маршруты часто объявляются как JSX-компоненты (`<Route>`), а не как конфиг-объект (хотя объектный API тоже есть). |
| SFC (`.vue`: template/script/style) | `.tsx`/`.jsx`-файл целиком на JS | Нет автоматической изоляции стилей «из коробки» — нужен отдельный механизм (CSS Modules, styled-components и т.п.). |
| `defineProps<T>()` / `defineEmits<T>()` | `interface Props {}` в сигнатуре функции-компонента | Нет компиляторных макросов — обычные TS-типы, ничего специального. |
| директивы (`v-bind`, `v-on`, кастомные) | атрибуты JSX = обычные JS-выражения и обработчики-пропы (`onClick={...}`) | Нет отдельного синтаксиса «директив» — всё едино с обычным JS-выражением. |
| точечная реактивность (обновляется только то, что реально изменилось) | рендер всей функции-компонента целиком при каждом обновлении state, затем сверка (reconciliation) с реальным DOM | Кажется, что «весь компонент перерисовывается — это медленно». На практике сверка обычно достаточно быстрая, но это меняет то, как вы думаете про мемоизацию и оптимизацию. |

## Мини-проверка себя

Ответьте прежде, чем откроете ответы ниже.

1. Почему `state.items.push(newItem)` не обновит экран в React, хотя аналогичный код с `reactive()` в Vue работает?
2. Чем `useEffect` принципиально отличается от `watch`, помимо синтаксиса?
3. Что произойдёт, если вызвать хук внутри `if`?
4. Почему `useRef` не вызывает перерисовку компонента при изменении `.current`?
5. Чем именованный слот Vue отличается от того, как в React передают JSX через пропы?

<details>
<summary>Ответы</summary>

1. React не отслеживает мутации объектов/массивов — он сравнивает, изменилась ли ссылка на state между рендерами. `push` мутирует существующий массив на месте, ссылка не меняется, поэтому React не видит повода перерисовывать компонент. Нужно передать в сеттер новый массив, например `setItems([...items, newItem])`.
2. `watch` подписывается на конкретный реактивный источник и запускается автоматическим отслеживанием зависимостей. `useEffect` не «подписывается» ни на что — он выполняется после рендера всегда, когда изменилось хотя бы одно значение из явно указанного массива зависимостей. Это описание синхронизации с внешним миром, а не реакция на конкретное изменение.
3. Нарушится порядок и количество хуков между рендерами — React связывает состояние хука с его порядковым номером вызова, а не с именем. Результат — некорректная работа или ошибка в рантайме. Условную логику нужно писать внутри хука, а не вокруг его вызова.
4. Потому что `useRef` специально спроектирован не участвовать в цикле рендера — это просто мутируемая коробка, которая переживает рендеры. Если бы изменение `.current` вызывало перерисовку, `useRef` ничем не отличался бы от `useState`.
5. Именованный слот — часть шаблонной системы Vue: родитель заполняет именованные секции разметки, а компонент решает, где их вставить, через `<slot name="...">`. В React нет отдельного механизма — эффект эмулируют обычным пропом, значением которого является JSX-элемент (например, `<Modal header={<CustomHeader />}>`), и рендерят его как любое другое значение пропа.

</details>
