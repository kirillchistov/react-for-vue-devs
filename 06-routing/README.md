# Модуль 06. Роутинг и модальные маршруты

## Смена со штурманом Роутером

«Здесь я вас не буду мучить — прокладка курса между картами станции у React Router работает почти так же, как у Vue Router: те же `<Route>`, тот же `useParams`, тот же `useNavigate` вместо `router.push`. Переучиваться почти нечему.

Кроме одного манёвра, которым я горжусь: показать соседнюю луну в боковом окне, не покидая орбиты. На человеческом языке — открыть детали ингредиента как модалку, но при этом URL меняется, страницу можно обновить (F5) и получить её уже целиком, а если поделиться ссылкой — по ней откроется именно то, что вы видели. Ни один известный мне Vue-роутер такого не делает из коробки».

## Идея модуля: обычный роутинг — знакомый, модальный маршрут — новый

### Что не меняется

`<Routes>`/`<Route>`, `useParams()`, `<Link>`/`<NavLink>` — концептуально то же самое, что `<router-view>`, `route.params`, `<router-link>` в Vue Router. Здесь этому не посвящается отдельная теория — переносите интуицию как есть.

### Модальный маршрут: URL меняется, а фон не размонтируется

Задача: клик по карточке должен открыть детали ингредиента как модалку (URL — `/ingredients/:id`), но каталог за спиной должен остаться на экране, со всем, что там было — поиском, прокруткой, состоянием. При этом если обновить страницу прямо на `/ingredients/:id` или перейти по прямой ссылке — должна открыться полноценная страница, без каталога за спиной, потому что открывать модалку не поверх чего.

Решение — паттерн «фоновый location» (**background location**):

```jsx
function App() {
  const location = useLocation();
  const background = location.state?.background; // «а был ли переход изнутри?»

  return (
    <>
      {/* рендерим маршруты ТАК, будто адрес всё ещё background — если он есть */}
      <Routes location={background || location}>
        <Route path="/" element={<HomePage />} />
        <Route path="/ingredients/:id" element={<IngredientPage />} />
      </Routes>

      {/* а поверх — ещё один набор маршрутов по НАСТОЯЩЕМУ текущему URL */}
      {background && (
        <Routes>
          <Route path="/ingredients/:id" element={<Modal><IngredientDetailsRoute /></Modal>} />
        </Routes>
      )}
    </>
  );
}
```

Ключевой момент — при переходе на карточку вы не используете `<Link to="/ingredients/123">` напрямую, а вызываете `navigate` с явным state:

```js
navigate(`/ingredients/${id}`, { state: { background: location } });
```

Если такого перехода не было (прямая ссылка, обновление страницы), `location.state` пуст, `background` — `undefined`, и первый блок `<Routes>` рендерит маршруты по настоящему текущему URL — то есть `IngredientPage` целиком, без каталога и без модалки.

## Инспектор Тай уже здесь

```ts
interface NavigationState {
  background?: Location;
}

const background = (location.state as NavigationState | null)?.background;
```

`location.state` в React Router типизирован как `unknown` — библиотека принципиально не может знать, что именно вы туда положите. Приведение типа (`as NavigationState | null`) здесь не формальность: это то место, где вы явно документируете форму данных, которую сами же туда кладёте несколькими файлами ранее (`navigate(path, { state: { background: location } })`). Если забудете обновить один из двух концов — TypeScript не поймает рассинхронизацию сам, потому что `unknown` намеренно отключает проверку. Это осознанный компромисс библиотеки, а не недосмотр.

## Проверь себя

<details>
<summary>1. Пользователь кликает карточку ингредиента, видит модалку, копирует ссылку из адресной строки и открывает её в новой вкладке. Что он увидит?</summary>

Полноценную страницу с деталями ингредиента, без каталога за спиной и без модального окна. В новой вкладке `location.state` пуст (state — часть истории браузера конкретной вкладки, а не самого URL), значит `background` окажется `undefined`, и приложение отрендерит `IngredientPage` как обычный, самостоятельный маршрут.
</details>

<details>
<summary>2. Почему для перехода к деталям используют <code>navigate(path, { state: { background: location } })</code>, а не просто <code>&lt;Link to={path}&gt;</code>?</summary>

`<Link>` тоже умеет принимать `state` (через проп `state`), так что дело не в этом — просто `navigate` явно показывает, что переход complet с побочным эффектом (передачей текущего `location` как фона), а не просто сменой адреса. Ключевая часть — именно `state: { background: location }`: без него на другом конце (в `App`) неоткуда узнать, что переход случился «изнутри» приложения, а не по прямой ссылке.
</details>

<details>
<summary>3. Зачем второй <code>&lt;Routes&gt;</code> рендерится БЕЗ проп <code>location</code>, в отличие от первого?</summary>

Первый `<Routes location={background || location}>` намеренно рендерит маршруты по фоновому (старому) адресу, чтобы каталог не размонтировался. Второй блок должен реагировать на настоящий текущий URL — именно по нему видно, что мы сейчас на `/ingredients/:id` и пора показать модалку. Если бы у него тоже был `location={background}`, он бы никогда не увидел переход на страницу ингредиента.
</details>

<details>
<summary>4. <code>IngredientDetailsRoute</code> используется и в модалке, и в отдельной странице <code>IngredientPage</code>. Зачем выносить его в отдельный компонент, а не продублировать логику поиска ингредиента в обоих местах?</summary>

Это модуль 04 в действии: одна и та же логика (взять `id` из `useParams`, дождаться `useIngredients`, найти нужный ингредиент) нужна в двух разных обёртках — где-то как модалка, где-то как самостоятельная страница. Компонент, не привязанный к конкретному способу показа, можно переиспользовать в обеих ситуациях, вместо того чтобы писать одну и ту же логику дважды и потом чинить баг в двух местах вместо одного.
</details>

<details>
<summary>5. Почему <code>NavLink</code> на <code>"/"</code> получает проп <code>end</code>, а остальные — нет?</summary>

По умолчанию `NavLink` считает себя активным, если текущий путь начинается с его `to` — а с `to="/"` совпадает вообще любой путь в приложении. `end` включает точное совпадение: ссылка на `"/"` активна, только когда путь — это ровно `"/"`, а не `"/feed"` или `"/profile"`, которые тоже технически «начинаются с /».
</details>
